# Qidra production readiness checklist

Дата аудита: 2026-06-06.

Цель: подготовить платформу к работе с реальными клиентами и USDT-операциями, закрыть критические баги перед деплоем и зафиксировать, что еще должно быть настроено вне кода.

## 1. Клиентские сценарии

- [x] Регистрация и вход имеют уведомления после ключевых действий.
- [x] Новые пароли и сброс пароля требуют усиленную password policy.
- [x] Google OAuth и Telegram OAuth добавлены как способы входа.
- [x] Заблокированный пользователь не должен получать активную сессию.
- [x] В личном кабинете авторизованный пользователь видит `Выход`, а не `Вход`.
- [x] Автовыход по бездействию настроен на 10 минут на клиентской стороне.
- [x] Серверная JWT-сессия ограничена 8 часами с обновлением не чаще 15 минут.
- [x] Чувствительные API защищены базовым rate limit на уровне приложения.
- [x] Mutating `/api/*` запросы блокируют чужой browser Origin/Referer.
- [x] KYC требует документы и сохраняет предыдущее значение, если участник обновляет только часть анкеты.
- [x] Заявка на размещение проекта требует одобренный KYC.
- [x] Заявка на размещение проекта требует минимум один документ.
- [x] Проектная заявка принимает несколько документов.
- [x] Проектная заявка ограничивает массовую загрузку: до 20 файлов и до 100 МБ суммарно.

Критично перед продом:

- [x] Добавлен адаптер постоянного файлового хранилища для KYC и документов проектов: local для разработки, S3-compatible private bucket для production.
- [ ] Подключить production `FILE_STORAGE_DRIVER=s3` и приватный bucket до запуска с реальными клиентами.
- [ ] Провести тест регистрации, восстановления пароля и email-уведомлений на реальном SMTP-домене.
- [ ] Проверить OAuth callback URLs на production-домене.

## 2. Кошелек, депозиты, выводы и участие

- [x] У каждого участника есть персональный TRC20-адрес.
- [x] QR-код и кнопка копирования адреса доступны на странице кошелька.
- [x] Ручное подтверждение входящих USDT-депозитов админом запрещено.
- [x] Входящий депозит подтверждается только после сверки hash, суммы, сети и персонального адреса.
- [x] Дублирующий или чужой transaction hash не должен зачисляться повторно.
- [x] Участие в проекте списывает только проверенный доступный баланс.
- [x] Заявка на участие выше доступного баланса блокируется.
- [x] При недостатке баланса система возвращает сумму, которую нужно пополнить.
- [x] Участие сразу активируется из проверенного баланса без ручного подтверждения админа.
- [x] Вывод средств требует одобренный KYC. Исправлено в этом аудите.
- [x] Участие разрешено только в активных опубликованных проектах. Исправлено в этом аудите.
- [x] Участие не может превысить оставшийся целевой объем проекта. Исправлено в этом аудите.
- [x] Лимит оставшегося объема проекта защищен от одновременных заявок. Исправлено в этом аудите.

Критично перед продом:

- [ ] Установить production `TRONGRID_API_KEY`, `QIDRA_USDT_TRC20_CONTRACT`, `QIDRA_WALLET_KEY_ENCRYPTION_SECRET`, `QIDRA_TRON_WALLET_ADDRESS`.
- [ ] Провести dry-run с маленькой реальной суммой USDT TRC20: депозит, автосверка, участие, админ-аудит, вывод.
- [ ] Настроить ротацию и хранение приватных ключей персональных адресов.
- [ ] Настроить мониторинг cron-синхронизации депозитов.

## 3. Админ-панель, роли и безопасность

- [x] У супер-админа есть доступ к пользователям, KYC, платежам, проектам, заявкам, журналу действий и поддержке.
- [x] Роли менеджеров разделены: главный админ, админ, техподдержка, отдел продаж, участник.
- [x] Изменение ролей требует причины и подтверждения `CONFIRM`.
- [x] Нельзя снять роль с последнего главного админа.
- [x] Супер-админ может временно или постоянно блокировать пользователя.
- [x] Блокировка требует причины, подтверждения и записи в журнал.
- [x] Нельзя заблокировать собственный аккаунт.
- [x] В карточке клиента доступны профиль, KYC, документы, платежи, участие, аудит и восстановление доступа.
- [x] Восстановление доступа через поддержку отправляет ссылку, а не задает пароль вручную.
- [x] В админ-панель добавлена страница диагностики production-безопасности без раскрытия секретов.

Критично перед продом:

- [ ] Сменить все локальные/тестовые пароли админов.
- [ ] Включить 2FA для админов и сотрудников, если production-окружение поддерживает `ENABLE_2FA`.
- [ ] Добавить резервного главного админа до запуска.
- [ ] Зафиксировать внутренний регламент: кто может корректировать баланс, кто подтверждает вывод, кто закрывает KYC.

## 4. Проекты, листинг и документы

- [x] Участник может подать собственный проект после KYC.
- [x] При выборе отрасли `Другое` требуется указать собственную отрасль.
- [x] В заявке проекта есть описание, целевой объем, ожидаемый результат и доходность.
- [x] Админ может взять заявку в проверку, отклонить или подготовить проект к листингу.
- [x] Создание проекта из заявки требует заполнения карточки проекта и подтверждения.
- [x] Карточки проектов показывают описание, целевой объем, модель, ожидаемый результат и доходность.
- [x] Документы проекта доступны инвестору через карточку проекта, если документы опубликованы.

Критично перед продом:

- [ ] Подготовить реальные юридические документы для каждого проекта.
- [ ] Проверить тексты доходности: не должно быть обещаний гарантированной прибыли без юридического основания.
- [ ] Настроить процесс финального approve перед публикацией карточки проекта.

## 5. Поддержка, коммуникации и аудит

- [x] Участник имеет вкладку поддержки.
- [x] Сотрудники видят очередь обращений по своим ролям.
- [x] Техподдержка и продажи разделяются по направлению обращения.
- [x] Чаты имеют статусы и историю сообщений.
- [x] Действия админов и сотрудников пишутся в журнал действий.
- [x] На админских страницах есть поиск клиента по имени или email.

Критично перед продом:

- [ ] Проверить уведомления о новых обращениях для сотрудников.
- [ ] Добавить SLA/статусы для очереди поддержки, если они должны быть обязательной частью процесса.
- [ ] Проверить сценарий оценки качества после закрытия обращения.

## Инфраструктура перед реальным запуском

- [ ] Production database с ежедневными backup и точкой восстановления.
- [ ] HTTPS production-домен.
- [ ] `NEXTAUTH_URL` равен production-домену.
- [ ] `NEXTAUTH_SECRET` сгенерирован заново и хранится только в secrets.
- [ ] Все OAuth redirect/callback URL добавлены для production.
- [ ] SMTP настроен на домене Qidra и проходит SPF/DKIM/DMARC.
- [ ] `CRON_SECRET` и `QIDRA_WALLET_SYNC_SECRET` заданы в secrets.
- [ ] `FILE_STORAGE_DRIVER=s3` подключен к приватному bucket с выключенным публичным доступом.
- [ ] Dependency audit разобран без `npm audit fix --force`; опасные транзитивные зависимости обновлены отдельным тестируемым планом.
- [x] Security headers включены в Next config: CSP без `unsafe-eval` в production, HSTS, frame protection, referrer policy, MIME sniffing protection, permissions policy.
- [x] Админ видит production-блокеры по secrets, SMTP, OAuth, S3 и TRC20 на странице `/admin/security`.
- [ ] Настроить внешний WAF/rate limit на уровне Cloudflare/Vercel/nginx для `/api/auth/*`, `/api/wallet/*`, `/api/investments`, `/api/support/*`.
- [ ] Логи ошибок и бизнес-аудит доступны администратору.
- [ ] Настроен rollback-план деплоя.
- [ ] Старые статические файлы `index.html`, `app.js`, `styles.css` не используются как production-приложение вместо Next.js.

## Итог аудита

Кодовая часть закрывает основные критические риски вокруг ручного присвоения депозитов, участия сверх баланса, участия в недоступном проекте, превышения объема сбора, вывода без KYC и административных действий без журнала.

Платформа еще не должна запускаться на реальные деньги, пока не закрыты внешние production-блокеры: production S3/R2-хранилище, production secrets, SMTP, OAuth callbacks, TronGrid production-настройки, backup базы, dependency audit, тестовый реальный USDT dry-run и регламент админских операций.
